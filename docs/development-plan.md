# 《我》内容优先开发计划

## 当前阶段目标

当前阶段不再以“先做第二章可玩 Demo”为第一目标，而是改为优先完成五章小说式母本，并基于母本建立后续游戏化拆解规则。

新的核心判断是：现有粗糙 demo 只能证明流程骨架可跑，不能证明《我》的情绪、人物弧线和章节节奏成立。继续优先推进 Cocos 灰盒或 Web 原型，会让开发被不具备可玩性的验证产物牵着走。因此，当前最重要的不是补 demo，而是先把完整内容骨架写对。

当前阶段的唯一内容基准文档为 [《我》五章小说式母本](./me-story-manuscript.md)。后续任何可玩切片、章节脚本、点击文本、谜题稿和 Demo 选章，都必须从该母本反拆，不再允许绕开母本另起一套结构。

## 文档分工

- [《我》创作蓝本](./me-creative-blueprint.md)：负责世界观、设计支柱、文本风格、章节意图、视觉与音频方向。
- [《我》五章小说式母本](./me-story-manuscript.md)：负责五章连续剧情正文、主角弧线、章节节奏、意象回收与现实回声。
- 本文档：负责当前阶段开发顺序、交付物、母本到游戏的提炼规则，以及 Demo 在整体路线中的降级处理。

`game/` 下现有 `game/src/gameState.js`、`game/web/app.js`、`game/assets/scripts/DemoStateBridge.ts` 等实现，暂时视为早期验证产物。它们可以保留用于回顾技术方向，但不再代表近期优先级，也不构成当前阶段的核心接口。

## 范围边界

当前阶段的工作范围严格限定为内容建设与内容到游戏的拆解方法，不直接推进以下事项：

- 不继续扩写或打磨现有第二章粗糙 demo。
- 不优先制作 Cocos 正式场景、灰盒关卡或小游戏 UI。
- 不先做 10-20 分钟可玩切片来倒逼剧情。
- 不在这一阶段接入平台能力、包体优化、小游戏适配、广告、商业化和发布流程。

这并不意味着旧 demo 文件需要删除，而是意味着它们在计划中的优先级被明确降级。是否保留、复用或废弃这些实现，放到母本完成并选定首个切片后再决定。

## 当前阶段交付物

本阶段交付物只有两项：

1. 一份完整的五章小说式母本文档。
2. 一份内容优先版开发计划文档。

完成标准不是“能运行”，而是：

- 五章主线完整闭环，不再只有设定与摘要。
- 每章都能清楚读出现实状态、入界动作、里世界经历、情绪转折和现实回声。
- 开发顺序已经从“demo 先行”改为“母本先行”。

## 母本到游戏的提炼规则

完成小说式母本后，后续游戏化拆解必须遵守以下方法：

### 章节结构提炼

每一章都应从母本中提炼出六类稳定单元：

1. 现实段：玩家进入本章前的现实生活瞬间，以及主角此刻的情绪底色。
2. 入界动作：那个可重复、可感知、能把现实推向里世界的动作。
3. 里世界子场景：从 prose 中拆出的 2-3 个主要探索空间。
4. 主谜题：本章最核心的一条“童年规则”式解法。
5. 隐藏道具：那些“没用但很酷”的收集物，以及它们对应的情绪意义。
6. 现实回声：主角回到现实后最小但清晰的变化。

如果某段内容无法归入这六类，也不必强行转成交互。母本中的留白、停顿、环境感和意象，有些应保留为氛围，而不是被过度功能化。

### 文本类型提炼

母本里的 prose 不能整体直接搬进游戏，而应按用途拆分：

- 可转为点击文本：那些针对树、路灯、课桌、柜门、车窗等具体物件的感知句。
- 可转为纸条或儿童痕迹：那些带规则感、提醒感、愿望感的短句。
- 可转为旁白：章节开场、转场、回归现实时的少量内心句。
- 可转为互动反馈：跑圈、翻页、躲藏、摆放声音、调整手影后即时得到的简短回应。

拆分时应保留“少而准”的表达原则，不把母本文字生硬塞成长对白或大段说明。

### 氛围与交互的边界

下列内容允许只作为氛围存在，不必勉强设计玩法：

- 影子、风、气味、旧纸张、粉笔灰、雨声等纯感受性意象。
- 用来承接章节情绪的停顿段落。
- 不承担解题功能、只负责让玩家感到被世界轻轻接住的片段。

只有当某个意象在母本中已经天然承担“规则、选择、通道、提示或回收”作用时，才优先转成交互或谜题。

### Demo 选章约束

未来如果重新进入 Demo 阶段，只能从母本中选一个章节或章节切片进行制作，不能绕开母本重新发明结构。首个切片是否仍为第二章，要等母本整体完成后，结合节奏、资源量和可玩性再决定。

## 新的阶段顺序

当前默认顺序固定为：

1. 完成五章小说式母本。
2. 明确母本到游戏的提炼规则。
3. 依据母本重排开发优先级与切片策略。
4. 再决定首个可制作章节或 Demo 切片。
5. 最后才进入 Cocos 场景、交互脚本、UI 和小游戏构建。

这个顺序的目的，是确保后续任何实现都建立在完整内容骨架之上，而不是继续围绕一个方向不稳的 demo 修修补补。

## 旧 demo 的当前定位

当前仓库内已有的 demo 相关实现，现阶段仅保留三种用途：

- 回顾技术验证方向是否可行。
- 作为未来正式切片的状态机参考。
- 帮助估算后来把章节拆成交互时需要哪些基础模块。

它们不再承担“证明作品方向成立”的职责。作品方向是否成立，首先由母本是否成立来判断。

## 近期检查清单

在开始下一阶段以前，应先确认以下几点：

- 母本文档已经覆盖五章完整主线，而不是重复创作蓝本摘要。
- 每章都具备现实开场、入界动作、里世界正文、情绪转折和现实回声。
- 本文档已经明确 demo 降级，不再把 Cocos 或 Web 原型列为第一优先级。
- 母本到游戏的提炼规则已经写清，后续实现不会绕开母本。

只有这几项成立后，才适合继续讨论第一个可玩切片怎么选、怎么做。
